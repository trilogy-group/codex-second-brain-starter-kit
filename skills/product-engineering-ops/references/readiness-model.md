# Engineering Readiness Model

## Purpose

Use this model to decide whether a product second brain is merely informative or actually usable for engineering work.

## Status Definitions

- `done`: the capability is in place, automated where appropriate, and evidenced
- `partial`: useful but incomplete, still dependent on manual work, or still product-specific
- `missing`: absent or too fragile to rely on

## Recommended Categories

### Ask 1: Import Any Product

- Reusable import factory
- Source ingestion and authenticated capture
- Manifest quality and validation
- Automation pack coverage

### Ask 2: Engineering-Ready Product System

- Code and repo intelligence
- Safe repo mirror sync
- Local runtime / feature development support
- Support and bug flow into code
- CI/CD intelligence
- Deployment and release intelligence

## Evidence Guidance

Good evidence:
- absolute file paths
- automation ids
- script paths
- PRs
- generated reports

Weak evidence:
- vague statements without an artifact
